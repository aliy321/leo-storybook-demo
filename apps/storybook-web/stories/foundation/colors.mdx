# Colors

LEO v3 uses **semantic color tokens** — role-based names like `primary`, `foreground`, and `muted` instead of visual descriptions like `canvas`, `inverse`, or `inverse-active`. The same hex values from Figma v2 are preserved; only the naming architecture changed.

Toggle **Brand** and **Scheme** in the Storybook toolbar to preview how tokens adapt across `default`, `agency`, and `takaful` brands in light and dark modes.

## Two independent axes

| Axis | Attribute | Values | What it controls |
|------|-----------|--------|------------------|
| **Brand** | `data-brand` (web) / Theme name (native) | `default`, `agency`, `takaful` | Which brand palette — primary red vs agency blue vs takaful orange |
| **Color scheme** | `data-color-scheme` (web) / color scheme (native) | `light`, `dark` | Light or dark appearance within a brand |

```html
<!-- Web: set both on <html> -->
<html data-brand="agency" data-color-scheme="dark">
```

Semantic tokens **auto-switch** when either axis changes. There is no manual "invert" toggle — scheme is not a user-facing action, it is a context attribute.

## How to read semantic token names

Semantic tokens describe **what the color is for**, not what it looks like:

| Token | Role | Tailwind examples |
|-------|------|-------------------|
| `background` | Page / app shell surface | `bg-background` |
| `foreground` | Default text on background | `text-foreground` |
| `primary` | Brand CTA, main actions | `bg-primary`, `text-primary` |
| `primary-foreground` | Text on primary surfaces | `text-primary-foreground` |
| `secondary` | Secondary actions | `bg-secondary` |
| `secondary-foreground` | Text on secondary surfaces | `text-secondary-foreground` |
| `muted` | Subtle / de-emphasized surfaces | `bg-muted` |
| `muted-foreground` | De-emphasized text | `text-muted-foreground` |
| `accent` | Hover / highlight surfaces | `bg-accent` |
| `accent-foreground` | Text on accent surfaces | `text-accent-foreground` |
| `destructive` | Error / danger actions | `bg-destructive` |
| `destructive-foreground` | Text on destructive surfaces | `text-destructive-foreground` |
| `border` | Default borders | `border-border` |
| `input` | Form field borders | `border-input` |
| `ring` | Focus rings | `ring-ring` |
| `card` | Elevated surfaces (cards, modals) | `bg-card` |
| `card-foreground` | Text on card surfaces | `text-card-foreground` |
| `popover` | Floating surfaces | `bg-popover` |
| `popover-foreground` | Text on floating surfaces | `text-popover-foreground` |
| `success` | Successful status | `bg-success` |
| `success-foreground` | Text on successful status | `text-success-foreground` |
| `warning` | Warning status | `bg-warning` |
| `warning-foreground` | Text on warning status | `text-warning-foreground` |
| `info` | Informational status | `bg-info` |
| `info-foreground` | Text on informational status | `text-info-foreground` |

```tsx
// All utilities bind to the same CSS variable
<View className="bg-primary" />
<Text className="text-primary" />
<View className="border border-primary" />
```

## Primitive colors

Primitive colors are **static hex values** that do not change with theme. Use them for illustrations, charts, and custom graphics — not for standard UI surfaces.

> Prefer semantic colors for UI. Primitives do not adapt to light/dark modes or brands.

## Best practices

### Do

- Use semantic tokens for all standard UI (`bg-primary`, `text-foreground`, `border-border`)
- Pair muted backgrounds with their semantic text (`bg-muted` + `text-muted-foreground`)
- Test across all three brands and both schemes using the Storybook toolbar
- Use primitive colors only for non-UI graphics

### Don't

- Don't invent aliases outside `component-tokens.json`
- Don't hardcode hex when a semantic token exists
- Don't use primitive palette names for theme-aware surfaces
